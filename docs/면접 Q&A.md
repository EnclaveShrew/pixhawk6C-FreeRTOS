# FC/임베디드 포트폴리오 면접 Q&A

---

## 1. SPI 통신

**Q. SPI Mode(CPOL/CPHA)를 어떻게 선택했나?**
데이터시트 타이밍 다이어그램에서 클럭 유휴 상태(CPOL)와 데이터 캡처 엣지(CPHA)를 확인. ICM-42688-P, BMI055 모두 Mode 0/3 지원 → Mode 3 선택. 같은 버스 공유 시 모드 통일 필요.

**Q. SPI 버스에 센서 3개를 연결했을 때 CS 관리를 어떻게 했나?**
센서마다 독립 GPIO를 CS로 할당. 부팅 시 모든 CS를 HIGH로 먼저 설정하여 버스 충돌 방지. 통신 시 해당 CS만 LOW → 전송 → HIGH. 미선택 센서는 MISO가 High-Z(고임피던스)로 버스에서 분리됨. spi_wrapper에서 spi_dev_t 구조체로 CS 핀을 추상화하여 드라이버가 특정 핀에 종속되지 않게 설계.

**Q. CS가 HIGH일 때 센서의 MISO는 어떤 상태인가? 왜 그래야 하나?**
High-Z. GND에도 VCC에도 연결 안 된 상태라 버스에 영향을 주지 않음. 선택된 센서만 MISO를 구동하므로 충돌 없음.

**Q. Full-Duplex인데 읽기 시 dummy byte를 보내는 이유는?**
SPI 클럭은 마스터의 전송 동작으로만 생성됨. 읽기 시에도 마스터가 뭔가를 보내야 클럭이 나가고, 그래야 슬레이브가 데이터를 내보낼 수 있음. dummy(0x00)는 클럭 생성용.

**Q. 같은 SPI 버스에 최대 클럭이 다른 센서가 있으면 어떻게 하나?**
느린 쪽에 맞추거나, CS 전환 시 SPI prescaler를 동적으로 변경. Pixhawk 6C에서는 IMU가 별도 SPI 버스에 있어 각각 최적 클럭 사용 가능.

## 2. I2C 통신

**Q. SPI와 I2C의 차이점은? 왜 IMU는 SPI, 바로미터/자력계는 I2C를 쓰나?**
SPI: 4라인, 고속(수십MHz), Full-Duplex → IMU처럼 1kHz 고속 읽기에 적합.
I2C: 2라인(SDA/SCL), 저속(~400kHz), Half-Duplex, 주소로 슬레이브 구분 → 바로미터/자력계처럼 저속+핀 절약이 유리한 경우.

**Q. I2C에서 같은 주소의 센서 2개를 연결하려면 어떻게 하나?**
I2C 주소가 같으면 같은 버스에 못 넣음. 방법: (1) 센서의 주소 핀(A0 등)으로 변경 (2) 별도 I2C 버스 사용 (3) I2C MUX 칩 사용.

**Q. I2C에서 RE-START 조건이 뭐고, 왜 필요한가?**
레지스터 읽기 시 필요. 주소+W로 레지스터 주소를 보낸 뒤, STOP 없이 바로 START를 다시 걸어서 주소+R로 전환. STOP을 걸면 다른 마스터가 버스를 뺏을 수 있어서 RE-START로 원자적으로 처리.

## 3. UART

**Q. UART는 SPI/I2C와 달리 클럭 라인이 없는데, 어떻게 동기화하나?**
송수신 양쪽이 미리 보레이트를 합의. 수신측은 START 비트(HIGH→LOW)를 감지하면 그 시점부터 보레이트 간격으로 비트를 샘플링. 보레이트 오차 ±2% 이내여야 안정적 통신 가능.

**Q. UART 보레이트 정확도는 어떻게 계산하나?**
`보레이트 = UART클럭 / (16 × USARTDIV)`. USARTDIV가 정수로 떨어지면 오차 0%. 안 떨어지면 반올림에 의한 오차 발생. 예: 115200bps에서 0.16% 오차.

**Q. 하드웨어 플로우컨트롤(RTS/CTS)은 왜 쓰나?**
텔레메트리 라디오가 무선 채널 상태에 따라 속도 변동. FC가 너무 빨리 보내면 라디오 버퍼 오버플로우. CTS로 "잠깐 멈춰"를 알림. 소프트웨어 플로우컨트롤(XON/XOFF)은 바이너리 데이터(MAVLink)와 충돌 가능하여 하드웨어 방식 사용.

## 4. 하드웨어 추상화 / 드라이버 설계

**Q. 래퍼(wrapper)를 왜 두었나?**
폴링→DMA 전환 시 래퍼 내부만 수정하고 센서 드라이버는 그대로. MCU 포팅 시에도 래퍼만 재작성. 테스트 시 mock으로 교체 가능.

**Q. MS5611은 왜 I2C 래퍼를 안 쓰고 HAL을 직접 호출하나?**
MS5611은 일반적인 레지스터 R/W가 아니라 커맨드 전송→변환 대기→ADC 읽기 방식. i2c_read_reg/write_reg 패턴에 안 맞아서 HAL_I2C_Master_Transmit/Receive를 직접 사용.

**Q. 폴링 vs 인터럽트 vs DMA의 장단점은?**
폴링: 구현 쉬움, CPU 블록. 인터럽트: 바이트마다 ISR 진입, 중간 수준. DMA: CPU 점유 최저, 하지만 구현 복잡(채널 할당, 메모리 정렬, STM32H7에서는 캐시 coherency 문제).

**Q. 에러가 발생하면 어떻게 처리했나?**
계층별로 자기 수준의 에러코드 반환. HAL_TIMEOUT → 래퍼에서 -1 → 드라이버에서 SENSOR_ERR_COMM → 호출자에서 이중 IMU 전환 등 대응. 상위 레이어는 하위 구현 상세를 몰라도 됨. 규칙: 0=성공, 음수=에러.

**Q. 센서 초기화 시 readback 검증을 하는 이유는?**
SPI/I2C 쓰기가 성공(HAL_OK)해도 실제로 센서에 값이 안 쓰였을 수 있음(배선 문제, 센서 고장 등). 쓴 뒤 다시 읽어서 일치 확인하면 확실히 설정이 적용됐다고 보장.

**Q. DMA로 전환할 때 래퍼 외에 뭘 더 고려해야 하나?**
STM32H7의 D-Cache가 DMA 버퍼를 캐싱하면 CPU가 읽는 값과 DMA가 쓴 값이 다를 수 있음. 대응: 버퍼를 non-cacheable 영역에 배치하거나 캐시 invalidation 수행. DMA 채널/스트림 할당 계획, 메모리 정렬(워드 바운더리)도 필요.

## 5. 클럭 / 전원 설계

**Q. 클럭 트리를 직접 설계한 경험이 있나? PLL 설정 근거는?**
HSE 16MHz → PLL1: DIVM=2, DIVN=120, DIVP=2 → SYSCLK 480MHz. PLL1Q=240MHz → SPI 커널 클럭. PLL3Q → USB 48MHz. VOS0(Overdrive) 활성화 필수. Flash latency 4 wait state.

**Q. PLL1P, PLL2Q 같은 이름의 의미는?**
숫자=PLL 번호, 알파벳=그 PLL 내부 출력 분주기(P/Q/R). PLL 하나에 3개 분주기가 있어서 하나의 VCO에서 서로 다른 클럭 3개 생성 가능. 어떤 페리페럴에 연결되느냐는 MCU 내부 MUX로 결정.

**Q. STM32H7의 클럭 도메인(D1/D2/D3)이 왜 분리되어 있나?**
D1(CPU), D2(페리페럴), D3(GPIO/저전력) 독립 전력 관리. 저전력 모드에서 일부 도메인만 활성 가능. 페리페럴마다 커널 클럭을 개별 선택 가능(H7 특징).

**Q. 타이머 클럭 x2 규칙이 뭔가?**
APB 프리스케일러가 1이 아니면 타이머 클럭 = APB × 2. PWM 주파수 계산, 보레이트 계산에 영향. 예: APB1=120MHz, 프리스케일러=2 → TIM 클럭=240MHz.

**Q. VOS0(Overdrive)는 뭐고 왜 필요한가?**
STM32H743가 480MHz로 동작하려면 필수. VOS1(400MHz) → ODEN 비트 설정 → VOS0(480MHz). 전력 소비 최대.

**Q. 센서 전원 시퀀싱은 왜 필요한가?**
MCU가 먼저 부팅 → GPIO 초기화(CS=HIGH) → 센서 전원 ON(VDD_3V3_SENSORS_EN) → 안정화 대기 → 센서 초기화. 순서가 반대면 CS가 Floating 상태에서 센서 전원이 켜져 버스 충돌 발생.

## 6. GPIO / 핀 설계

**Q. GPIO 초기값 설정이 왜 중요한가?**
리셋 직후 GPIO는 Input Floating. CS 핀이 LOW로 읽히면 모든 센서 동시 선택→버스 충돌. Active-Low 전원 핀이 LOW면 의도치 않은 전원 ON→돌입전류. CubeMX에서 초기값을 안전 상태로 설정해야 함.

**Q. GPIO Speed 설정이 EMI와 어떤 관계가 있나?**
Speed가 높으면 슬루레이트(신호 전환 속도)가 빨라져 EMI 증가. 필요 이상으로 높이면 크로스토크 발생. SPI CLK→High 이상, CS/LED→Low로 충분.

**Q. EXTI(외부 인터럽트)를 센서 DRDY와 어떻게 연동하나?**
센서 DRDY 핀 → MCU EXTI → ISR에서 태스크 노티피케이션으로 센서 읽기 태스크를 깨움. ISR에서 직접 SPI 전송하면 안 됨(시간 너무 김). EXTI 라인은 포트별 공유(PA0/PB0 같은 EXTI0 동시 불가).

## 7. 센서 특화

**Q. BMI055의 CS 핀이 2개인 이유는? 코드에서 어떻게 처리했나?**
가속도계(BMA280)와 자이로(BMG160)가 별도 다이. 각각 독립 CS 필요. bmi055_dev_t에 spi_dev_t 2개(accel_dev, gyro_dev)를 묶어서 처리. 버스트 읽기 불가(SPI 2회 필요).

**Q. 이중 IMU를 왜 쓰고, 두 센서 출력이 다르면 어떻게 하나?**
약간의 차이: 센서 특성 차이로 정상. 큰 차이: 최근 N샘플 분산 비교→분산 큰 쪽이 이상. 완전 고장(통신 실패): 즉시 나머지 센서로 전환. 다른 제조사(InvenSense vs Bosch)를 쓰는 이유는 공통 모드 고장 방지.

**Q. IMU의 스케일 팩터(LSB/g, LSB/dps)는 어떻게 결정되나?**
데이터시트에 FS(Full Scale) 설정별 감도가 명시됨. ±16g → 2048 LSB/g (ICM) 또는 128 LSB/g (BMI055). raw × (9.80665 / LSB_per_g) = m/s². 자이로도 동일 방식으로 dps→rad/s 변환.

**Q. 바로미터에서 2nd order temperature compensation이 뭐고 왜 필요한가?**
MS5611의 압전 소자는 온도에 따라 출력이 변함. 1차 보상만으로는 저온에서 오차가 커서, 20°C 이하에서 추가 보정(2차), -15°C 이하에서 한 번 더 보정(3차). 데이터시트에 공식이 정수 연산으로 제공됨.

**Q. MS5611의 PROM 교정 계수(C1~C6)는 뭔가?**
공장에서 개별 센서를 교정하여 ROM에 저장한 값. C1=압력 감도, C2=압력 오프셋, C3~C4=온도 계수, C5=기준 온도, C6=온도 감도. 초기화 시 읽어서 raw ADC 값을 실제 온도/기압으로 변환하는 데 사용.

**Q. IST8310에서 Single measurement 모드를 쓴 이유는?**
매번 측정 트리거(CNTL1에 0x01 쓰기) → DRDY 대기 → 데이터 읽기. 자력계는 수십Hz면 충분하고, Continuous 모드는 불필요한 전력 소비. Single 모드로 필요할 때만 측정하면 전력 절약 + 읽기 타이밍 제어 가능.

**Q. 자력계에서 hard/soft iron calibration이 뭔가?**
Hard iron: 주변 자석(모터, 배터리)이 만드는 고정 오프셋. 측정값에 상수를 빼서 보정.
Soft iron: 주변 금속이 자기장을 왜곡. 원 형태여야 할 XY 플롯이 타원이 됨. 행렬 곱으로 타원→원으로 보정. 둘 다 비행 전 캘리브레이션 루틴으로 측정.

## 8. GPS / UBX 프로토콜

**Q. UBX 파서를 상태 머신으로 구현한 이유는?**
UART는 바이트가 연속으로 들어옴. 패킷 경계를 모르는 상태에서 바이트마다 상태를 전이하며 패킷을 조립하는 게 가장 자연스러움. 버퍼에 모아서 파싱하려면 "어디까지 모아야 하나" 판단이 필요한데, 그게 결국 상태 머신. 메모리도 적게 씀.

**Q. Fletcher-8 체크섬은 어떻게 동작하나?**
CLASS부터 PAYLOAD 끝까지 바이트를 순회하며 ck_a += byte, ck_b += ck_a를 누적. 단순 합보다 바이트 순서 오류도 감지 가능. UBX 패킷 끝의 CK_A/CK_B와 비교하여 무결성 검증.

**Q. NMEA 대신 UBX를 쓴 이유는?**
NMEA: 텍스트 기반("$GPGGA,...")→파싱 복잡, 대역폭 낭비. UBX: 바이너리→파싱 효율적, 같은 정보를 적은 바이트로 전달. NAV-PVT 하나로 위치/속도/시간/위성 수 전부 포함.

**Q. GPS에서 fix type(2D/3D)의 차이는?**
2D: 위성 3개, 위도/경도만 확정, 고도 불확실. 3D: 위성 4개 이상, 위도/경도/고도 모두 확정. FC에서는 3D fix가 되어야 고도 기반 비행 가능.

## 9. AHRS (자세 추정)

**Q. AHRS가 뭔가? 물리적 장치인가?**
소프트웨어 알고리즘. 센서 데이터(자이로+가속도계+자력계)를 융합해서 현재 기체의 자세(roll, pitch, yaw)를 추정. 물리적으로 존재하지 않고, MCU에서 연산으로 돌아감.

**Q. 상보 필터의 원리는?**
자이로(단기 정확, 장기 드리프트)와 가속도계(장기 안정, 단기 노이즈)를 섞어서 양쪽 장점만 취함. `자세 = α × 자이로 예측 + (1-α) × 가속도계 측정`. α=0.98이면 자이로를 98% 믿고, 가속도계가 매 루프 2%씩 드리프트를 보정.

**Q. EKF와 상보 필터의 차이는?**
상보 필터: α 하나로 단순 융합, 자이로 바이어스 보정 못 함. EKF: 예측→보정 2단계, 센서별 노이즈 모델링(Q, R 행렬), 자이로 바이어스를 상태 벡터에 포함시켜 자동 추정/보정. 정밀도는 EKF가 높지만 구현 복잡.

**Q. EKF 상태 벡터에 자이로 바이어스를 포함시킨 이유는?**
자이로는 온도, 시간에 따라 바이어스(고정 오차)가 변함. 상태 벡터에 넣으면 EKF가 매 루프 바이어스를 추정하고 자이로 입력에서 자동 제거. `ω_corrected = ω_measured - bias_estimated`. 상보 필터에서는 이게 불가능해서 드리프트가 남음.

**Q. 쿼터니언을 사용한 이유는? 오일러 각도와 차이?**
오일러(roll/pitch/yaw)는 pitch ±90°에서 짐벌 락 발생(회전축 겹침, 자유도 상실). 쿼터니언(w,x,y,z)은 4개 숫자로 특이점 없이 3D 회전 표현. 회전 합성이 곱셈 한 번으로 효율적. 내부는 쿼터니언, 사용자 출력만 오일러로 변환.

**Q. 쿼터니언 정규화를 왜 하나?**
유효한 회전 쿼터니언은 `w²+x²+y²+z²=1`이어야 함. 매 루프 적분하면서 부동소수점 오차가 누적되면 1에서 벗어남. 주기적으로 정규화해서 1로 유지. 안 하면 자세 표현이 왜곡됨.

**Q. 가속도계로 yaw를 구할 수 없는 이유는?**
가속도계는 중력 방향만 측정. 중력은 항상 아래를 가리키므로 기울기(roll, pitch)는 알 수 있지만, 수평면 내 회전(yaw)은 중력과 무관해서 구분 불가. yaw 보정에는 자력계(지구 자기장 방향) 또는 GPS가 필요.

**Q. tilt-compensated 자력계 yaw란?**
자력계를 수평 상태에서 읽으면 yaw가 정확하지만, 기체가 기울면 자기장 벡터도 기울어져 yaw가 틀어짐. 현재 roll/pitch로 자력계 벡터를 수평면으로 회전 보정한 뒤 yaw를 계산하는 것.

**Q. 칼만 필터에서 P 행렬(공분산)의 의미는?**
현재 상태 추정의 불확실성. 예측 단계에서 P 증가(불확실해짐), 측정 업데이트 단계에서 P 감소(확신 높아짐). 칼만 게인 K는 P에서 자동으로 결정 — 불확실할수록 측정값을 많이 신뢰, 확실할수록 예측을 유지.

**Q. PX4/ArduPilot은 어떤 AHRS를 쓰나?**
둘 다 EKF 단독 사용(EKF2/EKF3). 상태 ~24개로 자세+위치+속도+바이어스+바람까지 추정. 우리 EKF는 자세+바이어스 7개만 추정하는 간소화 버전. 교육/연구 목적에 충분.

## 10. 제어기 (PID / LQR / INDI)

**Q. 캐스케이드 PID의 구조는? 왜 자세+각속도 2중 루프인가?**
외부 루프(자세 P): 자세 오차 → 목표 각속도. 내부 루프(각속도 PID): 각속도 오차 → 제어 출력. 자세만 직접 PID하면 각속도 응답이 느려서 진동 발생. 내부에 빠른 각속도 루프를 두면 자이로의 고속 피드백으로 안정성 확보.

**Q. Anti-windup이 뭔가? 왜 필요한가?**
PID 적분기가 오차를 계속 누적하면 출력이 포화(saturation)된 상태에서도 적분값이 계속 커짐. 이후 오차 부호가 바뀌어도 큰 적분값이 사라지는 데 시간이 걸려 오버슈트 발생. 적분값에 상한/하한을 두어(클램핑) 방지.

**Q. Derivative filtering은 왜 하나?**
미분항은 오차 변화율을 계산하는데, 센서 노이즈가 있으면 미분값이 크게 튀어서 출력이 진동함. LPF(저역통과 필터)를 적용하여 고주파 노이즈를 억제. `d_filtered += alpha * (d_raw - d_filtered)`.

**Q. 자세 오차를 쿼터니언으로 어떻게 계산하나?**
`q_err = q_current⁻¹ × q_target`. 단위 쿼터니언의 역=켤레(w,-x,-y,-z). 오차 벡터 = `2 × sign(q_err.w) × [q_err.x, q_err.y, q_err.z]`. sign 반전은 최단 경로 회전 보장용(359°가 아닌 1° 회전).

**Q. LQR이 PID와 다른 점은?**
PID: 각 축 독립 제어, 게인 수동 튜닝. LQR: 6차원 상태(자세 오차 3 + 각속도 오차 3)를 동시에 고려, 비용함수를 최소화하는 최적 게인 K를 수학적으로 계산. `u = -K × x`. 축 간 커플링도 K 행렬에 반영 가능. 단점: 선형화된 모델이 필요하고, 적분기가 없어 정상상태 오차 남을 수 있음.

**Q. LQR의 Q, R 행렬은 뭔가?**
Q: 상태 오차 가중치. Q가 크면 오차를 빨리 줄이려 함(빠른 응답, 공격적). R: 제어 입력 가중치. R이 크면 모터를 아끼려 함(느린 응답, 부드러움). Q/R 비율이 제어 성격을 결정.

**Q. INDI가 PID/LQR보다 외란에 강한 이유는?**
PID/LQR: 모델 기반으로 필요 토크를 계산 → 모델과 실제 차이(바람, 무게 변화 등)가 오차로 남음. INDI: `Δu = G⁻¹ × (ω̇_desired - ω̇_measured)`, 이전 출력에 증분만 더함. 실제 각가속도를 측정해서 피드백하므로 모델 오차/외란이 자동 상쇄. 단점: 각가속도 측정이 노이즈에 민감(LPF 필수).

**Q. INDI에서 G 행렬(control effectiveness)은 뭔가?**
각 축의 제어 입력이 각가속도에 미치는 영향. `ω̇ = G × u`. G ≈ J⁻¹ (관성 모멘트의 역수). 대각 근사하면 축 간 독립으로 단순화. 실제 값은 기체 파라미터에서 결정.

**Q. 세 제어기의 장단점 비교?**

| | PID | LQR | INDI |
|---|---|---|---|
| 모델 필요도 | 없음 | 높음 (선형화) | 낮음 (G만) |
| 튜닝 | 수동 (경험) | 수학적 최적 | G + 필터 |
| 외란 강인성 | 보통 | 보통 | 높음 |
| 구현 복잡도 | 낮음 | 중간 | 중간 |
| 적분 보정 | 있음 (I항) | 없음 | 없음 |
| 축 간 커플링 | 무시 | 반영 가능 | 무시 (대각 근사 시) |

**Q. 제어기 런타임 교체를 어떻게 구현했나?**
controller_interface.h에 함수 포인터 테이블(init, update, set_param, reset) 정의. 각 제어기가 이 인터페이스를 구현한 전역 인스턴스를 제공. 교체 시 포인터만 바꾸면 동일한 update() 호출로 다른 알고리즘이 실행됨. 교체 전 reset()으로 이전 상태 초기화.

## 11. 모터 믹싱

**Q. 모터 믹싱이 뭔가?**
제어기 출력(roll, pitch, yaw, thrust)을 개별 모터 출력으로 변환. 예: roll 명령이 들어오면 한쪽 모터는 세게, 반대쪽은 약하게. `motor[i] = thrust + mix[i][0]×roll + mix[i][1]×pitch + mix[i][2]×yaw`.

**Q. 쿼드콥터 X 배치에서 yaw는 어떻게 제어하나?**
대각선 모터 쌍의 회전 방향이 다름. CW 모터 쌍과 CCW 모터 쌍의 출력 차이로 반토크(reaction torque)를 만들어 yaw 회전. 모터 자체의 반토크를 이용하므로 별도 요 제어면(러더)이 필요 없음.

**Q. idle throttle은 왜 필요한가?**
모터가 완전히 멈추면 다시 가속하는 데 시간이 걸림(ESC 응답 지연). 최소 회전수를 유지하면 제어 응답성이 빨라짐. 또한 급강하 시에도 자세 제어를 유지할 수 있음. 보통 5% 정도.

## 12. 소프트웨어 설계 패턴

**Q. 함수 포인터 테이블 패턴은 뭔가?**
C에서 다형성(polymorphism)을 구현하는 방법. 구조체에 함수 포인터를 모아두고, 같은 인터페이스로 다른 구현을 호출. C++의 가상 함수 테이블(vtable)과 유사. 이 프로젝트에서 controller_t가 이 패턴.

**Q. static 변수로 모듈 상태를 관리하는 이유는?**
전역 변수 대신 파일 스코프 static을 쓰면 다른 파일에서 접근 불가(캡슐화). 각 제어기(pid, lqr, indi)가 자기 상태를 static으로 갖고 있어서 외부에서 직접 수정 불가. API(init, update, set_param)를 통해서만 상호작용.

**Q. void* config 패턴의 장단점은?**
장점: 공통 인터페이스(controller_t)에서 제어기별로 다른 설정 구조체를 받을 수 있음. 단점: 타입 안전성 없음(잘못된 구조체 전달해도 컴파일 에러 안 남). C에서 제네릭을 구현하는 일반적인 방법. 호출 측이 올바른 타입을 전달할 책임.

## 13. FC 시스템 레벨

**Q. FMU와 IO Processor의 역할 차이와 페일세이프 구조는?**
FMU(H743): 센서, AHRS, 제어 연산. IO(F103): PWM 출력, 페일세이프. FMU가 크래시하면 IO가 감지하고 안전 동작(서서히 하강 또는 RC 패스스루) 실행.

**Q. PWM ESC 제어에서 타이머 PSC/ARR 설정 근거는?**
TIM 클럭 240MHz, PSC=239→카운터 1MHz(1us 단위), ARR=2499→주기 2500us=400Hz. CCR=1000~2000으로 펄스폭 제어. DShot 같은 디지털 프로토콜은 노이즈에 강하고 양방향 통신 가능.

**Q. FreeRTOS에서 NVIC 우선순위와 API 호출 제한은?**
configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY(보통 5) 이상 숫자(=낮은 우선순위)에서만 FreeRTOS FromISR API 호출 가능. 그보다 높은 우선순위(0~4)에서 호출하면 크리티컬 섹션이 깨져 크래시.

**Q. 센서 5개를 동시에 읽어야 할 때 어떻게 스케줄링하나?**
FreeRTOS 태스크 우선순위로 분리. IMU 1kHz(최고), AHRS 1kHz, 제어 1kHz, 바로미터/자력계 50~100Hz(중간), GPS 10Hz(낮음). 우선순위 역전 방지를 위해 태스크 노티피케이션 + 더블 버퍼링 활용.

**Q. 센서 통신 중 타임아웃이 발생하면 시스템 전체에 어떤 영향이 있고, 어떻게 대처하나?**
에러 카운터 증가, 이중 IMU면 전환, MAVLink 경고 전송. 일정 횟수 이상 실패 시 해당 센서 비활성화하고 주기적 복구 시도. 전체 시스템은 워치독 타이머로 보호.

## 14. 제어 이론 심화

**Q. Feedforward가 뭐고, PID에 왜 추가하나?**
PID는 오차가 생긴 "뒤에" 반응하는 피드백 제어. Feedforward는 목표값이 바뀔 때 "미리" 예상 출력을 넣어주는 것. 예: 목표 각속도가 100deg/s로 바뀌면, 그걸 만드는 데 필요한 토크를 모델에서 계산해서 바로 넣어줌. PID는 남은 오차만 보정. 결과적으로 응답이 빨라지고 오버슈트가 줄어듦. `출력 = FF(목표) + PID(오차)`.

**Q. Derivative kick이 뭔가? 어떻게 방지하나?**
목표값이 갑자기 바뀌면(스텝 입력), 오차가 한 샘플 만에 크게 변해서 D항이 스파이크를 만듦. 모터에 순간 충격이 감. 방지법: 오차를 미분하는 대신 **측정값만 미분**. `D = -Kd × (measurement - prev_measurement) / dt`. 목표값 변화는 D항에 안 들어가므로 스파이크 없음. PX4도 이 방식 사용.

**Q. Anti-windup에 클램핑 말고 다른 방법이 있나?**
Back-calculation. 출력이 포화되면, 포화된 만큼의 차이를 적분기에서 역으로 빼줌. `I -= Kb × (u_saturated - u_unsaturated)`. 클램핑보다 부드럽고, 포화에서 빠져나오는 속도를 Kb로 조절 가능. 상용 FC에서는 클램핑이 대부분 — 단순하고 인증에 유리.

**Q. 연속 시간 PID를 디지털로 구현할 때 주의점은?**
연속 시간 적분(∫)을 이산 합산(Σ)으로 바꿔야 함. 방법: (1) 직사각형 적분(Euler) `I += e × dt` — 가장 단순, (2) 사다리꼴 적분(Tustin) `I += (e + e_prev) / 2 × dt` — 더 정확. 미분도 차분으로 근사. dt가 충분히 작으면(1kHz) Euler로 충분. dt가 크면(50Hz 바깥 루프) Tustin이 안전.

**Q. Bumpless transfer(범프 없는 전환)가 뭔가?**
제어기를 런타임에 교체할 때, 새 제어기의 내부 상태(적분기 등)가 0이면 출력이 갑자기 바뀌어서 기체가 흔들림. 전환 시 새 제어기의 적분기를 현재 출력값에 맞게 초기화하면 출력이 연속적으로 이어짐. controller_t의 reset()에 이 로직을 넣을 수 있음.

**Q. Gain scheduling이 뭔가?**
비행 조건(고도, 속도, 배터리 전압 등)에 따라 PID 게인을 바꾸는 기법. 호버링 게인과 고속 비행 게인이 다를 수 있음. 미리 여러 조건에서 게인을 튜닝해두고, 현재 조건에 맞는 게인을 보간(interpolation)해서 적용.

**Q. PID 수동 튜닝 절차는?**
1. I, D를 0으로 놓고 P만 올림
2. P를 올려서 진동 시작 직전까지 → 임계 게인(Ku)
3. 그 상태에서 진동 주기 측정 → Tu
4. Ziegler-Nichols 공식으로 초기값 계산: Kp=0.6Ku, Ki=2Kp/Tu, Kd=KpTu/8
5. 이후 수동 미세 조정
6. **핵심:** 안쪽 루프(각속도)를 먼저 튜닝 → 바깥 루프(자세) 순서. 바깥 먼저 하면 안쪽이 불안정해서 의미 없음.

**Q. 안정성(stability)을 어떻게 판단하나?**
실전적으로: 스텝 응답에서 (1) 발산하지 않는지, (2) 진동이 감쇠하는지, (3) 정착 시간이 합리적인지. 이론적으로: 폐루프 전달함수의 극점이 모두 좌반면에 있으면 안정. LQR은 수학적으로 안정성이 보장되는 게 장점. PID는 게인 잘못 잡으면 발산 가능.

**Q. 모터 출력이 포화(saturation)되면 어떻게 처리하나?**
4개 모터 중 하나가 100%를 넘으면 그 모터만 자르면 자세 제어가 깨짐. 우선순위: 자세 제어(roll/pitch/yaw) > 추력(thrust). 방법: 전체 모터 출력을 비례 스케일링하여 범위 안에 넣되, 자세 제어 차이는 유지. 또는 추력을 줄여서 자세 제어 여유를 확보.

## 15. FreeRTOS 심화

**Q. Task vs ISR 컨텍스트의 차이는?**
ISR: 하드웨어 인터럽트로 진입, 스택이 별도(MSP), 선점 불가(같은/낮은 우선순위), 짧게 끝내야 함. Task: 스케줄러가 관리, 각자 스택(PSP), 블로킹 가능(vTaskDelay 등). ISR에서는 FromISR 버전 API만 사용 가능. ISR에서 긴 작업 하면 다른 인터럽트/태스크가 굶음.

**Q. Priority Inversion이 뭔가? 어떻게 방지하나?**
고우선순위 태스크 H가 뮤텍스를 기다리는데, 저우선순위 L이 뮤텍스를 갖고 있고, 중간 우선순위 M이 L을 선점해서 결과적으로 H가 M보다 늦게 실행되는 상황. 방지: **Priority Inheritance** — L이 뮤텍스를 잡은 동안 H의 우선순위를 일시적으로 상속받아서 M에게 선점 안 당함. FreeRTOS에서 xSemaphoreCreateMutex()가 이걸 지원.

**Q. Mutex vs Binary Semaphore vs Counting Semaphore 차이는?**
Mutex: 소유권 있음(잠근 태스크만 풀 수 있음), priority inheritance 지원, 상호 배제용. Binary Semaphore: 소유권 없음, ISR에서 Give하고 태스크에서 Take하는 동기화/신호 전달용. Counting Semaphore: 여러 개의 리소스를 카운팅. 센서 DRDY → 태스크 깨우기에는 Binary Semaphore 또는 Task Notification이 적합.

**Q. Task Notification이 Semaphore보다 나은 점은?**
FreeRTOS Task Notification은 별도 커널 객체 생성 불필요, RAM 절약, 속도 ~45% 빠름. 1:1 신호 전달에 최적. 단점: 브로드캐스트 불가(1개 태스크에만 보낼 수 있음), 큐처럼 데이터를 여러 개 버퍼링 못 함. 센서 DRDY → 특정 태스크 깨우기에 딱 맞음.

**Q. FreeRTOS 스택 오버플로우를 어떻게 감지하나?**
configCHECK_FOR_STACK_OVERFLOW를 1 또는 2로 설정. 방법 1: 컨텍스트 스위칭 시 스택 포인터가 범위를 벗어났는지 확인. 방법 2: 스택 끝에 패턴(0xA5A5A5A5)을 써두고 훼손됐는지 확인. 방법 2가 더 확실. uxTaskGetStackHighWaterMark()로 여유 스택 확인 가능.

**Q. FreeRTOS Heap 할당 방식(heap_1~5)의 차이는?**
heap_1: malloc만 가능, free 불가. 가장 안전(단편화 없음), 초기화 시 한번 할당하고 끝나는 시스템에 적합. heap_2: free 가능하지만 단편화. heap_4: free 가능 + 인접 블록 병합(단편화 감소). heap_5: heap_4 + 여러 메모리 영역 지원(STM32H7의 DTCM/SRAM 영역 분리 시 유용).

**Q. Critical Section vs Mutex, 언제 뭘 쓰나?**
Critical Section(`taskENTER_CRITICAL`): 인터럽트를 끔. 매우 짧은 구간(수 us)에서만 사용. 길게 잡으면 인터럽트가 막혀서 센서 데이터 유실. Mutex: 스케줄러가 태스크를 블록시킴. 긴 작업(수 ms)에 적합. **규칙:** ISR↔태스크 동기화 = Task Notification/Semaphore, 태스크↔태스크 공유자원 = Mutex, 아주 짧은 공유변수 보호 = Critical Section.

**Q. 더블 버퍼링이 뭐고 왜 쓰나?**
버퍼 2개를 번갈아 사용. 센서 태스크가 A에 쓰는 동안 제어 태스크는 B를 읽음. 쓰기 완료되면 포인터만 스왑. 장점: 뮤텍스 없이 데이터 일관성 보장, 락 경합 없음, 실시간성 유지. 주의: 포인터 스왑은 원자적이어야 함(ARM Cortex-M에서 32비트 정렬 포인터 쓰기는 원자적).

**Q. vTaskDelay vs vTaskDelayUntil 차이는?**
vTaskDelay(10): "지금부터 10틱 쉬어라" → 태스크 실행 시간이 포함 안 되어서 실제 주기가 실행시간만큼 늘어남. vTaskDelayUntil: "다음 절대 시점까지 쉬어라" → 정확한 주기 보장. 1kHz 제어 루프처럼 정확한 주기가 필요한 곳에서는 반드시 vTaskDelayUntil 사용.

## 16. 임베디드 시스템 일반

**Q. volatile 키워드는 언제 써야 하나?**
컴파일러에게 "이 변수는 네가 모르게 바뀔 수 있다"고 알림. 필수인 경우: (1) ISR에서 수정하고 태스크에서 읽는 변수, (2) 하드웨어 레지스터(메모리 매핑 I/O), (3) DMA 버퍼. 안 붙이면 컴파일러가 최적화로 읽기를 생략해서 값이 안 바뀌는 것처럼 동작. 주의: volatile은 원자성을 보장하지 않음 — 32비트보다 큰 변수는 여전히 동기화 필요.

**Q. STM32의 메모리 맵(Flash/SRAM/DTCM 등)을 설명해라.**
STM32H743 기준: Flash(2MB, 0x0800_0000~) = 코드+상수. DTCM(128KB, 0x2000_0000~) = CPU 전용, 가장 빠름, 캐시 미경유, DMA 접근 불가. SRAM1~3(총 512KB+, 0x3000_0000~) = 일반 데이터, DMA 접근 가능. AXI SRAM(512KB, 0x2400_0000~) = 대용량 버퍼. 핵심: **DMA 버퍼는 DTCM에 두면 안 됨**(DMA가 접근 못 함). 반대로 실시간 제어 변수는 DTCM이 가장 빠름.

**Q. 링커 스크립트가 하는 일은?**
코드/데이터를 메모리의 어디에 배치할지 결정. `.text`(코드) → Flash, `.data`(초기화된 전역) → SRAM(Flash에서 복사), `.bss`(0 초기화 전역) → SRAM, 스택/힙 크기 지정. STM32H7에서는 DMA 버퍼를 특정 SRAM 영역에 배치하기 위해 커스텀 섹션을 만들기도 함.

**Q. 스타트업 코드(startup.s)에서 뭘 하나?**
리셋 벡터 진입 → (1) MSP(메인 스택 포인터) 초기화, (2) .data 섹션을 Flash에서 SRAM으로 복사, (3) .bss 섹션을 0으로 채움, (4) 시스템 클럭 초기화(SystemInit), (5) main() 호출. 인터럽트 벡터 테이블도 여기서 정의. CubeMX가 자동 생성하지만 원리를 알아야 디버깅 가능.

**Q. Endianness가 뭐고 센서 드라이버에서 왜 중요한가?**
바이트 저장 순서. Little-endian: 하위 바이트가 낮은 주소(ARM Cortex-M 기본). Big-endian: 상위 바이트가 낮은 주소. 센서마다 다름 — ICM-42688-P는 Big-endian(MSB first, `combine_bytes(high, low)`), BMI055는 Little-endian(LSB first, `combine_bytes_le(low, high)`). 잘못 조합하면 값이 엉뚱하게 됨. 데이터시트에서 바이트 순서를 반드시 확인.

**Q. STM32H7에서 D-Cache 문제가 뭔가?**
Cortex-M7에는 L1 D-Cache(16KB)가 있음. CPU가 SRAM을 읽으면 캐시에 복사본이 생김. 문제: DMA가 SRAM에 직접 쓰면 캐시에는 옛날 값이 남아 있어서 CPU가 DMA 결과를 못 봄. 해결: (1) DMA 완료 후 `SCB_InvalidateDCache_by_Addr()` 호출, (2) 버퍼를 non-cacheable 영역에 배치(MPU 설정), (3) DTCM 사용(캐시 안 거침, 단 DMA 접근 불가).

**Q. MPU(Memory Protection Unit)가 뭐고 왜 쓰나?**
메모리 영역별로 접근 권한(읽기/쓰기/실행), 캐시 정책을 설정. 용도: (1) 스택 오버플로우 시 HardFault로 잡기, (2) DMA 버퍼 영역을 non-cacheable로 설정, (3) 잘못된 메모리 접근 방지. STM32H7에서 D-Cache 문제 해결에 MPU가 핵심 역할.

**Q. HardFault가 발생하면 어떻게 디버깅하나?**
(1) SCB→CFSR 레지스터에서 원인 확인(Bus Fault, Usage Fault, MemManage Fault), (2) SCB→BFAR/MMFAR에서 문제 주소 확인, (3) 스택에서 복귀 주소(PC) 확인 → 어떤 코드에서 발생했는지 추적. 흔한 원인: 정렬되지 않은 메모리 접근, NULL 포인터, 스택 오버플로우.

## 17. PX4 아키텍처 (교양)

**Q. PX4의 uORB가 뭔가?**
Pub/Sub 메시지 버스. 모듈 간 직접 함수 호출 없이, 토픽에 publish하면 subscribe한 모듈이 데이터를 받음. 장점: 모듈 간 결합도 낮음(독립적 개발/테스트 가능). 단점: 함수 호출보다 오버헤드 있음, 데이터 흐름 추적이 어려움. 이 프로젝트에서 직접 함수 호출 방식을 쓴 이유: 모듈 수가 적고, 오버헤드 최소화 + 데이터 흐름 명확화.

**Q. PX4의 부팅 시퀀스를 설명해라.**
NuttX 부팅 → 파일시스템 마운트(SD카드) → rcS 스크립트 실행 → 파라미터 로드 → 센서 드라이버 시작 → EKF 시작 → 제어기 모듈 시작 → MAVLink 시작 → Commander(상태 머신) 시작. 각 단계가 NuttX 태스크로 실행.

**Q. NuttX vs FreeRTOS 차이는?**
NuttX: POSIX-like API(open/read/write), 파일시스템 내장, 네트워크 스택 내장, 상대적으로 무거움. FreeRTOS: 최소한의 스케줄러+동기화만 제공, 가벼움, 파일시스템/네트워크는 별도 추가. FC 전용 펌웨어에는 FreeRTOS가 더 적합하지만, PX4는 모듈형 구조와 파라미터/로깅 시스템을 위해 NuttX의 파일시스템이 필요.

**Q. PX4 파라미터 시스템은 어떻게 동작하나?**
`PARAM_DEFINE_FLOAT(MC_ROLLRATE_P, 0.15f)` 같은 매크로로 파라미터를 선언. SD카드에 저장/로드. QGC에서 실시간 변경 가능. MAVLink PARAM_SET 메시지로 수신 → 내부 파라미터 테이블 업데이트.

**Q. PX4 Commander 모듈의 역할은?**
기체 상태 머신 관리. DISARMED → ARMED → TAKING_OFF → FLYING → LANDING → DISARMED. 페일세이프 판단(RC 로스트, 배터리 저전압, GPS 로스트). 센서 상태 확인(Preflight Check). 모드 전환(Manual, Altitude, Position, Mission 등).

**Q. PX4의 EKF2와 이 프로젝트 EKF의 차이는?**
PX4 EKF2: 상태 24개(자세+속도+위치+자이로바이어스+가속도바이어스+자력계바이어스+바람속도 등). 다중 센서 융합. 이 프로젝트 EKF: 상태 7개(쿼터니언+자이로바이어스). 자세 추정에 집중한 간소화 버전.

## 18. MAVLink 프로토콜 (Phase 5 대비)

**Q. MAVLink v1과 v2의 차이는?**
v1: 헤더 6바이트, 메시지 ID 1바이트(256개 제한), 서명 없음. v2: 헤더 10바이트, 메시지 ID 3바이트(1600만개), 패킷 서명(인증) 지원, 빈 필드 트리밍(대역폭 절약). v2는 v1 하위호환. 새 프로젝트는 v2 사용이 표준.

**Q. MAVLink 메시지 직렬화 방식은?**
Little-endian, 필드를 크기 내림차순으로 정렬(8→4→2→1바이트). 이유: 패딩 없이 자연 정렬. CRC는 CRC-16/MCRF4XX + CRC_EXTRA(메시지 정의 해시)로 버전 불일치 감지.

**Q. Heartbeat 메시지가 왜 중요한가?**
MAVLink 시스템의 "살아있음" 신호. 1Hz로 전송. 기체 타입, 오토파일럿 타입, 현재 모드, 시스템 상태 포함. GCS는 Heartbeat가 일정 시간 안 오면 연결 끊김으로 판단.

**Q. MAVLink에서 신뢰성 있는 전송은 어떻게 하나?**
MAVLink 자체는 비신뢰성(UDP류). 중요한 메시지는 Command/Ack 패턴 사용. COMMAND_LONG 전송 → COMMAND_ACK 응답 대기 → 타임아웃 시 재전송. 미션 업로드는 핸드셰이크 프로토콜.

**Q. MAVLink 커스텀 메시지를 정의하려면?**
XML 파일에 메시지 정의 추가 → mavgenerate로 코드 생성. 개발용 ID 범위: 150000~. GCS(QGC)에서도 같은 XML로 빌드해야 인식.

## 19. CAN / DroneCAN (교양)

**Q. CAN 버스의 특징과 장점은?**
차동 신호(CANH/CANL)로 노이즈에 강함. 멀티마스터, 중재(arbitration)로 충돌 해결(ID 낮을수록 우선). 에러 감지/복구 내장. 최대 1Mbps. 배선이 2선으로 단순(종단저항 120Ω 필요). 드론에서는 ESC, GPS, 에어스피드 센서 등을 하나의 CAN 버스로 연결.

**Q. DroneCAN(구 UAVCAN)이 뭔가?**
CAN 버스 위에서 동작하는 드론 전용 프로토콜. 노드 ID, 메시지 타입, 서비스(Request/Response) 정의. 장점: PWM ESC보다 양방향 통신 가능(ESC가 RPM, 전류, 온도를 보고), 노이즈에 강함, 배선 단순.

## 20. DShot 프로토콜 (교양)

**Q. PWM ESC와 DShot ESC의 차이는?**
PWM: 아날로그 펄스폭(1000~2000us)으로 스로틀 전달. 노이즈에 취약, 단방향, 캘리브레이션 필요. DShot: 디지털 프로토콜(비트 단위 전송), CRC 검증, 캘리브레이션 불필요, 양방향(텔레메트리 ESC에서 RPM 피드백). DShot600이 현재 표준.

**Q. DShot 신호는 하드웨어적으로 어떻게 생성하나?**
타이머 PWM + DMA. 비트 1과 0을 서로 다른 듀티비의 PWM 펄스로 표현. 16비트 프레임(11비트 스로틀 + 1비트 텔레메트리 요청 + 4비트 CRC)을 DMA로 타이머 CCR에 연속 전송. CPU 개입 없이 정확한 타이밍 보장.

---

## 21. 비행 모드 / 상태 머신 (교양)

**Q. Manual / Stabilized / Altitude / Position 모드의 차이는?**
Manual: RC 입력이 모터에 직접 전달. Stabilized(Attitude): RC 스틱 = 목표 자세, FC가 자세 유지. Altitude: 스로틀은 목표 고도 변화율, FC가 바로미터로 고도 유지. Position: RC 스틱 = 목표 속도, FC가 GPS로 위치 유지. 위로 갈수록 센서 의존도 높음.

**Q. 페일세이프 시나리오와 대응을 설명해라.**
RC 로스트: 호버링 → 일정 시간 후 RTL(Return To Launch) → 배터리 한계 시 착륙. GPS 로스트: Altitude 모드로 전환. 배터리 저전압: 경고 → RTL → 강제 착륙. 센서 이상: 이중 IMU 전환, 전환 불가 시 비상 착륙.

## 22. 칼만 필터 수학 (필수)

**Q. 칼만 필터의 predict-update 2단계를 수식 없이 설명해라.**
Predict(예측): "이전 상태 + 모델(자이로)로 다음 상태를 예측. 불확실성(P)은 증가." Update(보정): "실제 측정값(가속도계)과 예측값을 비교. 둘의 불확실성 비율(칼만 게인 K)에 따라 가중 평균. 보정 후 불확실성(P)은 감소." 측정이 정확할수록 측정값을 많이 반영, 예측이 정확할수록 예측값을 유지.

**Q. 프로세스 노이즈(Q)와 측정 노이즈(R)는 뭔가?**
Q: 모델의 불확실성. "자이로 적분으로 자세를 예측하는데, 자이로 노이즈 때문에 예측이 얼마나 틀릴 수 있는가." Q가 크면 예측을 덜 신뢰 → 측정값 비중 증가. R: 측정의 불확실성. "가속도계 측정이 얼마나 노이즈인가." R이 크면 측정을 덜 신뢰 → 예측 비중 증가. Q/R 비율이 핵심이고, 데이터시트 노이즈 스펙에서 초기값을 잡고 실험으로 튜닝.

**Q. EKF에서 "Extended"가 의미하는 것은?**
기본 칼만 필터는 선형 시스템 전용. 쿼터니언 자세 역학은 비선형. EKF는 비선형 모델을 현재 상태 주변에서 1차 테일러 전개(야코비안)로 선형 근사하여 칼만 필터를 적용. 매 스텝 야코비안을 다시 계산하므로 연산량이 늘어남.

**Q. Innovation(잔차)이 뭐고 왜 중요한가?**
`innovation = 실제 측정값 - 예측 측정값`. EKF가 얼마나 잘 동작하는지의 지표. Innovation이 작으면 예측이 정확한 것. Innovation이 지속적으로 크면 모델이 틀렸거나 센서가 이상한 것.

## 23. 센서 신호처리 기초 (필수)

**Q. 로우패스 필터(LPF)의 원리는?**
고주파 성분을 감쇠시키고 저주파만 통과. 1차 IIR 디지털 LPF: `y = α × x + (1-α) × y_prev`. α가 작을수록 강하게 필터링(느리게 반응). 컷오프 주파수 fc와의 관계: `α = 2π×fc×dt / (1 + 2π×fc×dt)`. 센서 노이즈 제거, PID D항 필터링에 사용.

**Q. 노치 필터(Notch filter)는 언제 쓰나?**
특정 주파수만 골라서 제거. 드론에서 모터 진동 주파수가 알려져 있으면(예: 4000RPM → 66.7Hz), 그 주파수만 노치로 제거. LPF와 달리 넓은 주파수 대역을 손상시키지 않음. PX4/ArduPilot에서 IMU 노이즈 제거에 사용.

**Q. 샘플링 정리(나이퀴스트)가 뭐고 왜 알아야 하나?**
신호를 제대로 복원하려면 최대 주파수의 2배 이상으로 샘플링해야 함. IMU가 1kHz로 데이터를 내보내면 500Hz까지의 신호를 표현 가능. 500Hz 이상의 노이즈가 있으면 앨리어싱(저주파로 접혀서 가짜 신호 생성). 방지: 센서 내부 안티앨리어싱 필터 + 소프트웨어 LPF.

**Q. IMU의 ODR(Output Data Rate)과 BW(Bandwidth)의 관계는?**
ODR: 센서가 데이터를 내보내는 속도(1kHz면 1ms마다). BW: 센서 내부 LPF의 컷오프 주파수. BW가 ODR/2보다 높으면 앨리어싱 발생. ICM-42688-P는 ODR 1kHz에 BW 200Hz 정도가 적절.

## 24. 드론 하드웨어 일반 (교양)

**Q. ESC(Electronic Speed Controller)가 하는 일은?**
배터리 DC 전원을 3상 AC로 변환하여 브러시리스 모터를 구동. FC에서 스로틀 명령(PWM/DShot)을 받아 모터 회전수를 제어. 텔레메트리 ESC는 RPM, 전류, 온도를 FC에 피드백.

**Q. 브러시리스 모터의 KV가 뭔가?**
무부하 상태에서 1V당 RPM. KV=920이면 11.1V(3S 배터리)에서 약 10,212RPM. KV 높으면 고속/저토크(소형 프로펠러), KV 낮으면 저속/고토크(대형 프로펠러).

**Q. 프레임 진동이 제어에 어떤 영향을 주나?**
프레임 → 진동 → IMU에 전달 → 가속도계 노이즈 증가 → AHRS/EKF 정확도 저하 → 제어 품질 저하. 심하면 EKF 발산. 대응: 진동 격리 마운트(Pixhawk 내장), 소프트웨어 LPF/노치 필터, 프로펠러 밸런싱.

**Q. 배터리 전압 강하가 제어에 미치는 영향은?**
배터리가 방전되면 같은 PWM 듀티에서 모터 RPM이 떨어짐 → 실제 추력이 줄어듦. 보상 방법: 배터리 전압을 측정하여 제어 출력을 스케일링(voltage compensation).

## 25. 개발 도구 / 디버깅 (필수)

**Q. SWD(Serial Wire Debug)로 뭘 할 수 있나?**
ARM Cortex-M 디버그 인터페이스. 2핀(SWDIO, SWCLK). 브레이크포인트, 스텝 실행, 레지스터/메모리 실시간 관찰, Flash 프로그래밍. JTAG보다 핀 적음. ITM(SWO 핀)을 통한 printf 디버깅도 가능.

**Q. printf 디버깅 vs SWD 디버깅, 언제 뭘 쓰나?**
printf(UART/SWO): 시간 순서대로 흐름 추적에 좋음, 실시간 성능에 영향. SWD: 특정 시점의 상태를 깊게 분석, 브레이크 걸면 실시간 동작이 멈춤(제어 루프에서 위험). 실전: 개발 초기에 SWD로 드라이버 디버깅, 이후 printf/로깅으로 데이터 추적, 비행 테스트는 SD 로깅.

**Q. 비행 로그 분석은 어떤 도구로 하나?**
PX4: Flight Review(logs.px4.io). ArduPilot: MAVExplorer, Mission Planner. 범용: PlotJuggler. 확인 항목: EKF innovation, 진동 수준(FFT), PID 응답(목표 vs 실제), 배터리 전압/전류, GPS 정확도.

## 26. 안전 표준 / 인증 (교양 — 대형 무인기/UAM 타겟 시 중요)

**Q. DO-178C가 뭔가?**
항공 소프트웨어 인증 표준. 설계, 코딩, 검증, 형상관리 전 과정에 요구사항 정의. DAL A(가장 엄격)~DAL E(영향 없음). DAL A에서는 MC/DC 커버리지 100% 필요. 대형 무인기/UAM은 필수.

**Q. 왜 PID가 인증에 유리한가?**
DO-178C에서는 모든 동작을 "설명 가능"해야 함. PID: "오차에 비례하여 출력한다"로 설명 가능, 동작이 직관적. MPC: 최적화 솔버의 수렴성 증명이 필요, 인증 작업량 폭발. 인증 비용 때문에 상용 항공은 여전히 PID 기반 주류.

**Q. 이중화(Redundancy)의 종류는?**
Dual-Parallel: 두 채널이 동시에 계산, 출력 비교 → 불일치 시 하나 차단. Dual-Duplex: 두 채널이 교차 검증. Triplex: 3개 중 다수결(Voting). 소형 드론: 이중 IMU 정도. 대형 무인기: Dual-Duplex FCC + 페일오버. UAM급: Triplex + 안전 모듈.
