# FC/임베디드 포트폴리오 면접 Q&A

---

## 1. SPI 통신

**Q. SPI Mode(CPOL/CPHA)를 어떻게 선택했나?**
데이터시트 타이밍 다이어그램에서 클럭 유휴 상태(CPOL)와 데이터 캡처 엣지(CPHA)를 확인. ICM-42688-P, BMI055 모두 Mode 0/3 지원 → Mode 3 선택. 같은 버스 공유 시 모드 통일 필요.

**Q. SPI 버스에 센서 3개를 연결했을 때 CS 관리를 어떻게 했나?**
센서마다 독립 GPIO를 CS로 할당. 부팅 시 모든 CS를 HIGH로 먼저 설정하여 버스 충돌 방지. 통신 시 해당 CS만 LOW → 전송 → HIGH. 미선택 센서는 MISO가 High-Z(고임피던스)로 버스에서 분리됨. spi_wrapper에서 spi_dev_t 구조체로 CS 핀을 추상화하여 드라이버가 특정 핀에 종속되지 않게 설계.

**Q. CS가 HIGH일 때 센서의 MISO는 어떤 상태인가? 왜 그래야 하나?**
High-Z. GND에도 VCC에도 연결 안 된 상태라 버스에 영향을 주지 않음. 선택된 센서만 MISO를 구동하므로 충돌 없음.

**Q. Full-Duplex인데 읽기 시 dummy byte를 보내는 이유는?**
SPI 클럭은 마스터의 전송 동작으로만 생성됨. 읽기 시에도 마스터가 뭔가를 보내야 클럭이 나가고, 그래야 슬레이브가 데이터를 내보낼 수 있음. dummy(0x00)는 클럭 생성용.

**Q. 같은 SPI 버스에 최대 클럭이 다른 센서가 있으면 어떻게 하나?**
느린 쪽에 맞추거나, CS 전환 시 SPI prescaler를 동적으로 변경. Pixhawk 6C에서는 IMU가 별도 SPI 버스에 있어 각각 최적 클럭 사용 가능.

## 2. I2C 통신

**Q. SPI와 I2C의 차이점은? 왜 IMU는 SPI, 바로미터/자력계는 I2C를 쓰나?**
SPI: 4라인, 고속(수십MHz), Full-Duplex → IMU처럼 1kHz 고속 읽기에 적합.
I2C: 2라인(SDA/SCL), 저속(~400kHz), Half-Duplex, 주소로 슬레이브 구분 → 바로미터/자력계처럼 저속+핀 절약이 유리한 경우.

**Q. I2C에서 같은 주소의 센서 2개를 연결하려면 어떻게 하나?**
I2C 주소가 같으면 같은 버스에 못 넣음. 방법: (1) 센서의 주소 핀(A0 등)으로 변경 (2) 별도 I2C 버스 사용 (3) I2C MUX 칩 사용.

**Q. I2C에서 RE-START 조건이 뭐고, 왜 필요한가?**
레지스터 읽기 시 필요. 주소+W로 레지스터 주소를 보낸 뒤, STOP 없이 바로 START를 다시 걸어서 주소+R로 전환. STOP을 걸면 다른 마스터가 버스를 뺏을 수 있어서 RE-START로 원자적으로 처리.

## 3. UART

**Q. UART는 SPI/I2C와 달리 클럭 라인이 없는데, 어떻게 동기화하나?**
송수신 양쪽이 미리 보레이트를 합의. 수신측은 START 비트(HIGH→LOW)를 감지하면 그 시점부터 보레이트 간격으로 비트를 샘플링. 보레이트 오차 ±2% 이내여야 안정적 통신 가능.

**Q. UART 보레이트 정확도는 어떻게 계산하나?**
`보레이트 = UART클럭 / (16 × USARTDIV)`. USARTDIV가 정수로 떨어지면 오차 0%. 안 떨어지면 반올림에 의한 오차 발생. 예: 115200bps에서 0.16% 오차.

**Q. 하드웨어 플로우컨트롤(RTS/CTS)은 왜 쓰나?**
텔레메트리 라디오가 무선 채널 상태에 따라 속도 변동. FC가 너무 빨리 보내면 라디오 버퍼 오버플로우. CTS로 "잠깐 멈춰"를 알림. 소프트웨어 플로우컨트롤(XON/XOFF)은 바이너리 데이터(MAVLink)와 충돌 가능하여 하드웨어 방식 사용.

## 4. 하드웨어 추상화 / 드라이버 설계

**Q. 래퍼(wrapper)를 왜 두었나?**
폴링→DMA 전환 시 래퍼 내부만 수정하고 센서 드라이버는 그대로. MCU 포팅 시에도 래퍼만 재작성. 테스트 시 mock으로 교체 가능.

**Q. MS5611은 왜 I2C 래퍼를 안 쓰고 HAL을 직접 호출하나?**
MS5611은 일반적인 레지스터 R/W가 아니라 커맨드 전송→변환 대기→ADC 읽기 방식. i2c_read_reg/write_reg 패턴에 안 맞아서 HAL_I2C_Master_Transmit/Receive를 직접 사용.

**Q. 폴링 vs 인터럽트 vs DMA의 장단점은?**
폴링: 구현 쉬움, CPU 블록. 인터럽트: 바이트마다 ISR 진입, 중간 수준. DMA: CPU 점유 최저, 하지만 구현 복잡(채널 할당, 메모리 정렬, STM32H7에서는 캐시 coherency 문제).

**Q. 에러가 발생하면 어떻게 처리했나?**
계층별로 자기 수준의 에러코드 반환. HAL_TIMEOUT → 래퍼에서 -1 → 드라이버에서 SENSOR_ERR_COMM → 호출자에서 이중 IMU 전환 등 대응. 상위 레이어는 하위 구현 상세를 몰라도 됨. 규칙: 0=성공, 음수=에러.

**Q. 센서 초기화 시 readback 검증을 하는 이유는?**
SPI/I2C 쓰기가 성공(HAL_OK)해도 실제로 센서에 값이 안 쓰였을 수 있음(배선 문제, 센서 고장 등). 쓴 뒤 다시 읽어서 일치 확인하면 확실히 설정이 적용됐다고 보장.

**Q. DMA로 전환할 때 래퍼 외에 뭘 더 고려해야 하나?**
STM32H7의 D-Cache가 DMA 버퍼를 캐싱하면 CPU가 읽는 값과 DMA가 쓴 값이 다를 수 있음. 대응: 버퍼를 non-cacheable 영역에 배치하거나 캐시 invalidation 수행. DMA 채널/스트림 할당 계획, 메모리 정렬(워드 바운더리)도 필요.

## 5. 클럭 / 전원 설계

**Q. 클럭 트리를 직접 설계한 경험이 있나? PLL 설정 근거는?**
HSE 16MHz → PLL1: DIVM=2, DIVN=120, DIVP=2 → SYSCLK 480MHz. PLL1Q=240MHz → SPI 커널 클럭. PLL3Q → USB 48MHz. VOS0(Overdrive) 활성화 필수. Flash latency 4 wait state.

**Q. PLL1P, PLL2Q 같은 이름의 의미는?**
숫자=PLL 번호, 알파벳=그 PLL 내부 출력 분주기(P/Q/R). PLL 하나에 3개 분주기가 있어서 하나의 VCO에서 서로 다른 클럭 3개 생성 가능. 어떤 페리페럴에 연결되느냐는 MCU 내부 MUX로 결정.

**Q. STM32H7의 클럭 도메인(D1/D2/D3)이 왜 분리되어 있나?**
D1(CPU), D2(페리페럴), D3(GPIO/저전력) 독립 전력 관리. 저전력 모드에서 일부 도메인만 활성 가능. 페리페럴마다 커널 클럭을 개별 선택 가능(H7 특징).

**Q. 타이머 클럭 x2 규칙이 뭔가?**
APB 프리스케일러가 1이 아니면 타이머 클럭 = APB × 2. PWM 주파수 계산, 보레이트 계산에 영향. 예: APB1=120MHz, 프리스케일러=2 → TIM 클럭=240MHz.

**Q. VOS0(Overdrive)는 뭐고 왜 필요한가?**
STM32H743가 480MHz로 동작하려면 필수. VOS1(400MHz) → ODEN 비트 설정 → VOS0(480MHz). 전력 소비 최대.

**Q. 센서 전원 시퀀싱은 왜 필요한가?**
MCU가 먼저 부팅 → GPIO 초기화(CS=HIGH) → 센서 전원 ON(VDD_3V3_SENSORS_EN) → 안정화 대기 → 센서 초기화. 순서가 반대면 CS가 Floating 상태에서 센서 전원이 켜져 버스 충돌 발생.

## 6. GPIO / 핀 설계

**Q. GPIO 초기값 설정이 왜 중요한가?**
리셋 직후 GPIO는 Input Floating. CS 핀이 LOW로 읽히면 모든 센서 동시 선택→버스 충돌. Active-Low 전원 핀이 LOW면 의도치 않은 전원 ON→돌입전류. CubeMX에서 초기값을 안전 상태로 설정해야 함.

**Q. GPIO Speed 설정이 EMI와 어떤 관계가 있나?**
Speed가 높으면 슬루레이트(신호 전환 속도)가 빨라져 EMI 증가. 필요 이상으로 높이면 크로스토크 발생. SPI CLK→High 이상, CS/LED→Low로 충분.

**Q. EXTI(외부 인터럽트)를 센서 DRDY와 어떻게 연동하나?**
센서 DRDY 핀 → MCU EXTI → ISR에서 태스크 노티피케이션으로 센서 읽기 태스크를 깨움. ISR에서 직접 SPI 전송하면 안 됨(시간 너무 김). EXTI 라인은 포트별 공유(PA0/PB0 같은 EXTI0 동시 불가).

## 7. 센서 특화

**Q. BMI055의 CS 핀이 2개인 이유는? 코드에서 어떻게 처리했나?**
가속도계(BMA280)와 자이로(BMG160)가 별도 다이. 각각 독립 CS 필요. bmi055_dev_t에 spi_dev_t 2개(accel_dev, gyro_dev)를 묶어서 처리. 버스트 읽기 불가(SPI 2회 필요).

**Q. 이중 IMU를 왜 쓰고, 두 센서 출력이 다르면 어떻게 하나?**
약간의 차이: 센서 특성 차이로 정상. 큰 차이: 최근 N샘플 분산 비교→분산 큰 쪽이 이상. 완전 고장(통신 실패): 즉시 나머지 센서로 전환. 다른 제조사(InvenSense vs Bosch)를 쓰는 이유는 공통 모드 고장 방지.

**Q. IMU의 스케일 팩터(LSB/g, LSB/dps)는 어떻게 결정되나?**
데이터시트에 FS(Full Scale) 설정별 감도가 명시됨. ±16g → 2048 LSB/g (ICM) 또는 128 LSB/g (BMI055). raw × (9.80665 / LSB_per_g) = m/s². 자이로도 동일 방식으로 dps→rad/s 변환.

**Q. 바로미터에서 2nd order temperature compensation이 뭐고 왜 필요한가?**
MS5611의 압전 소자는 온도에 따라 출력이 변함. 1차 보상만으로는 저온에서 오차가 커서, 20°C 이하에서 추가 보정(2차), -15°C 이하에서 한 번 더 보정(3차). 데이터시트에 공식이 정수 연산으로 제공됨.

**Q. MS5611의 PROM 교정 계수(C1~C6)는 뭔가?**
공장에서 개별 센서를 교정하여 ROM에 저장한 값. C1=압력 감도, C2=압력 오프셋, C3~C4=온도 계수, C5=기준 온도, C6=온도 감도. 초기화 시 읽어서 raw ADC 값을 실제 온도/기압으로 변환하는 데 사용.

**Q. IST8310에서 Single measurement 모드를 쓴 이유는?**
매번 측정 트리거(CNTL1에 0x01 쓰기) → DRDY 대기 → 데이터 읽기. 자력계는 수십Hz면 충분하고, Continuous 모드는 불필요한 전력 소비. Single 모드로 필요할 때만 측정하면 전력 절약 + 읽기 타이밍 제어 가능.

**Q. 자력계에서 hard/soft iron calibration이 뭔가?**
Hard iron: 주변 자석(모터, 배터리)이 만드는 고정 오프셋. 측정값에 상수를 빼서 보정.
Soft iron: 주변 금속이 자기장을 왜곡. 원 형태여야 할 XY 플롯이 타원이 됨. 행렬 곱으로 타원→원으로 보정. 둘 다 비행 전 캘리브레이션 루틴으로 측정.

## 8. GPS / UBX 프로토콜

**Q. UBX 파서를 상태 머신으로 구현한 이유는?**
UART는 바이트가 연속으로 들어옴. 패킷 경계를 모르는 상태에서 바이트마다 상태를 전이하며 패킷을 조립하는 게 가장 자연스러움. 버퍼에 모아서 파싱하려면 "어디까지 모아야 하나" 판단이 필요한데, 그게 결국 상태 머신. 메모리도 적게 씀.

**Q. Fletcher-8 체크섬은 어떻게 동작하나?**
CLASS부터 PAYLOAD 끝까지 바이트를 순회하며 ck_a += byte, ck_b += ck_a를 누적. 단순 합보다 바이트 순서 오류도 감지 가능. UBX 패킷 끝의 CK_A/CK_B와 비교하여 무결성 검증.

**Q. NMEA 대신 UBX를 쓴 이유는?**
NMEA: 텍스트 기반("$GPGGA,...")→파싱 복잡, 대역폭 낭비. UBX: 바이너리→파싱 효율적, 같은 정보를 적은 바이트로 전달. NAV-PVT 하나로 위치/속도/시간/위성 수 전부 포함.

**Q. GPS에서 fix type(2D/3D)의 차이는?**
2D: 위성 3개, 위도/경도만 확정, 고도 불확실. 3D: 위성 4개 이상, 위도/경도/고도 모두 확정. FC에서는 3D fix가 되어야 고도 기반 비행 가능.

## 9. FC 시스템 레벨

**Q. FMU와 IO Processor의 역할 차이와 페일세이프 구조는?**
FMU(H743): 센서, AHRS, 제어 연산. IO(F103): PWM 출력, 페일세이프. FMU가 크래시하면 IO가 감지하고 안전 동작(서서히 하강 또는 RC 패스스루) 실행.

**Q. PWM ESC 제어에서 타이머 PSC/ARR 설정 근거는?**
TIM 클럭 240MHz, PSC=239→카운터 1MHz(1us 단위), ARR=2499→주기 2500us=400Hz. CCR=1000~2000으로 펄스폭 제어. DShot 같은 디지털 프로토콜은 노이즈에 강하고 양방향 통신 가능.

**Q. FreeRTOS에서 NVIC 우선순위와 API 호출 제한은?**
configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY(보통 5) 이상 숫자(=낮은 우선순위)에서만 FreeRTOS FromISR API 호출 가능. 그보다 높은 우선순위(0~4)에서 호출하면 크리티컬 섹션이 깨져 크래시.

**Q. 센서 5개를 동시에 읽어야 할 때 어떻게 스케줄링하나?**
FreeRTOS 태스크 우선순위로 분리. IMU 1kHz(최고), AHRS 1kHz, 제어 1kHz, 바로미터/자력계 50~100Hz(중간), GPS 10Hz(낮음). 우선순위 역전 방지를 위해 태스크 노티피케이션 + 더블 버퍼링 활용.

**Q. 센서 통신 중 타임아웃이 발생하면 시스템 전체에 어떤 영향이 있고, 어떻게 대처하나?**
에러 카운터 증가, 이중 IMU면 전환, MAVLink 경고 전송. 일정 횟수 이상 실패 시 해당 센서 비활성화하고 주기적 복구 시도. 전체 시스템은 워치독 타이머로 보호.
